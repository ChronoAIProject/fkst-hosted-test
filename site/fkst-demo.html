fkst demo

Watch a GitHub issue become a pull request

fkst treats GitHub as the control surface for autonomous software engineering. You file an issue, an isolated session claims the work, an LLM agent writes the code, and the result comes back as a pull request for the repository.

The delivery loop

The basic fkst path is intentionally small: GitHub issues are the queue, labels are the claims, and pull requests are the output. A repository with the fkst GitHub App installed can use normal issues to describe work, then let a session run that work in an isolated pod.

  1. File a work issue

    Describe the change in GitHub using the same level of detail you would give a human teammate. The issue carries the work label for the session that should claim it.

  2. Let the session claim it

    fkst runs a session from a trigger issue. The session watches for open issues with its work label and claims each delivery item so one isolated agent owns the task.

  3. Review the pull request

    The agent works in its sandbox, writes the change with an LLM coding agent, and opens a pull request. The repository owner keeps the normal review and merge step.

Issue-driven control

A fkst session starts from a trigger issue labelled fkst-substrate-trigger. That trigger issue names the packages to run, the session name, and the work label that identifies claimable issues. Keeping the control surface in GitHub makes queue state, ownership, and results visible where the team already works.

Sessions and work labels

A session is the running unit of delivery. It can stay active while labelled work remains, then idle when the queue is empty. Work labels let different sessions focus on different sets of issues, and each claimed item is handled by an isolated agent rather than shared process state.

Pull requests and merge policy

fkst returns completed work as pull requests. Depending on repository and session configuration, those pull requests can be reviewed manually or handled by an auto-merge flow. In either case, GitHub remains the durable record of the change and the issue it closes.

Key terms

Trigger issue
The GitHub issue labelled fkst-substrate-trigger that declares a session: its name, packages, and work label.
Session
The long-lived isolated agent that claims and works the labelled issues.
Work label
The label you add to an issue to route it to a session; while an open issue carries it, the session stays active.
Auto-merge
When enabled, the session's pull requests are merged automatically and the linked work issue is closed.
