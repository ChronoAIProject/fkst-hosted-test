Getting started

From a GitHub issue to a pull request, in four steps

Install the fkst GitHub App on a repository, open one fkst-substrate-trigger issue that names the packages to run and a work label, then file work issues under that label. fkst spawns one hard-isolated pod per session, an LLM agent writes the code, and you get pull requests back to review and merge. GitHub is the only state: issues are the queue, labels are the claims, PRs are the output.

Prerequisites

  • Admin access on the GitHub repository where fkst should work.
  • The fkst GitHub App installation URL from your operator.
  • One or more public package refs in the form owner/repo@ref:path.
  • Each package path contains an fkst.toml file.
  • An LLM endpoint configured by the operator; do not put keys in the issue.

Walkthrough

  1. Install the GitHub App

    Install the app on the repository where you want fkst to create pull requests. The app needs Issues read/write, Pull requests read/write, and Contents read/write so the control plane can reconcile that repository's trigger issues.

    Install the fkst App

  2. Open the trigger issue

    Create an issue labelled fkst-substrate-trigger. Its body must contain four ### sections: ### Session Name with one DNS-1123 label matching ^[a-z0-9]([a-z0-9-]*[a-z0-9])?$ and 1-40 characters; ### Packages with one public owner/repo@ref:path ref per line, each containing fkst.toml; ### Work Label with exactly one comma-free label up to 50 characters; and optional ### Environment with a named environment. 

    ### Session Name
devloop-demo

### Packages
ChronoAIProject/fkst-packages@dev:packages/github-devloop

### Work Label
agent-build

### Environment
node20

    A package ref follows this shape:

    owner/repo@ref:path/to/package

  3. File work issues under the label

    Every open issue with the work label is claimable. The session stays alive while at least one open labelled issue exists, idle-stops when the queue empties, and runs labelled issues concurrently. 

    Title: Add /healthz endpoint
Labels: agent-build

Body:
Add a /healthz endpoint that returns HTTP 200 with:

{"status":"ok"}

Include a test that verifies the status code and response body.

    Use multiple package refs when a session needs more than one package:

    ### Session Name
devloop-demo

### Packages
ChronoAIProject/fkst-packages@dev:packages/github-devloop
ChronoAIProject/fkst-packages@dev:packages/site

### Work Label
agent-build

### Environment
node20

  4. Review and merge the PRs

    The agent opens a pull request per issue. fkst manages lifecycle labels fkst-running, fkst-completed, fkst-terminated, and fkst-failed on the trigger issue, and comments on malformed input.

Troubleshooting

fkst couldn't parse this trigger issue: ...
A ### section is missing, duplicated, or malformed. Fix the four headings.
one or more ### Packages refs are not reachable on public GitHub
Check for a bad ref, private repository, or package path missing fkst.toml.
Invalid session name
Use a DNS-1123 label matching ^[a-z0-9]([a-z0-9-]*[a-z0-9])?$, 1-40 characters.
Work label rejected
Use exactly one line, keep it at 50 characters or fewer, and do not include a comma.
Nothing happens
Confirm the App is installed, the trigger issue has fkst-substrate-trigger, and at least one open issue carries the work label.
Environment not found
Create it first via PUT /api/v1/users/me/environments/{name} or omit the section.
