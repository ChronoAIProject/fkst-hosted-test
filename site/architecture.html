Architecture

One GitHub issue in, one pull request out — safely, concurrently.

fkst is an autonomous software-engineering platform. You file a GitHub issue; an isolated agent claims it, writes the code with an LLM, and opens a pull request. Three parts make that happen: a Lua engine that runs the work, a library of packages that define the work, and a cloud control plane that runs each job in its own hard-isolated Kubernetes pod. GitHub is the only durable state.

Three planes.

  • fkst-substrate — the engine

    Lua, event-driven.

    A Lua-based engine that runs packages as event-driven departments on a supervise loop. It reacts to events, drives each package's steps, and keeps the loop alive for the length of a session.

  • fkst-packages — the work

    The package library.

    The reusable units of behavior. github-devloop turns a GitHub issue into a pull request using the codex LLM agent; github-proxy polls GitHub for events; plus consensus and intake/decompose/pr/integration helpers that compose into a full delivery pipeline.

  • fkst-hosted — the control plane

    Rust · Axum · Kubernetes.

    The cloud service. When a repo has the fkst GitHub App installed, it reconciles open trigger issues and spawns exactly one hard-isolated pod per session running the substrate loop. It spawns and kills pods, rotates each pod's GitHub token, and serves a named-environments API.

Issue to pull request.

Pod-per-session, hard isolation.

Each delivery session runs in its own pod so concurrent work can proceed without sharing process, filesystem, or identity boundaries. The hosted control plane owns pod lifecycle; the agent owns only the repository work inside its session.

  • One pod per sessionno shared uid, filesystem, or PID namespace
  • No cluster accessno Kubernetes API and no mounted service account token
  • NetworkPolicy egressonly GitHub and the LLM endpoint
  • Dropped capabilitiesleast privilege inside the workload container
  • Run-to-completion + TTL GCpods are stopped and collected after completion or timeout

GitHub is the only durable state.

There is no database behind the delivery loop. Durable state stays where the work already lives, so queue position, ownership, and output are visible through normal GitHub objects.

The reconciler loop.

  1. Poll installed repositories for open trigger issues.
  2. Claim one issue by applying the ownership label.
  3. Spawn one pod with a short-lived GitHub App token.
  4. The pod runs the substrate loop; codex writes the change and opens a pull request.
  5. Rotate the token before expiry.
  6. On completion or stop, kill the pod and let TTL garbage collection remove it.

Token rotation keeps hard isolation and long sessions compatible: a pod never needs durable credentials, but the control plane can keep an authorized session alive long enough to finish real work.
